# Spec: Discogs Analytics Frontend — V1

## 1. Purpose

Build a minimal browser frontend for Demo Day that allows users to interact with the Discogs LangGraph analytics agent through a clean chat-style interface.

The frontend should make the project demoable by showing the full loop:

```text
Natural language query
  → FastAPI agent
  → LangGraph workflow
  → generated Python + DuckDB SQL
  → Plotly HTML artifact
  → rendered chart in browser
```

The frontend is not the core intelligence of the project. Its job is to make the agent easy to use, easy to demo, and visually clear.

---

## 2. Context

The backend agent already exposes, or is expected to expose, a FastAPI API that accepts natural language queries, runs the LangGraph agent, and returns:

- `thread_id`
- `run_id`
- assistant response
- chart artifact metadata
- generated SQL
- dataframe preview
- routing metadata
- validation metadata

The frontend must not access DuckDB, Postgres, ETL files, artifacts on disk, or LangGraph internals directly. It should only talk to the FastAPI backend.

---

## 3. V1 Goals

The frontend must:

- run in the browser;
- provide a query input box;
- send user queries to the backend `/query` endpoint;
- preserve the current `thread_id` to continue a conversation;
- allow starting a new conversation;
- show user and assistant messages;
- render Plotly HTML artifacts generated by the backend;
- show suggested demo queries;
- show generated SQL in a collapsible/debug panel;
- show dataframe preview when available;
- show route/model metadata when available;
- provide a clean demo-ready UI;
- be dockerized and runnable with the rest of the stack.

---

## 4. Non-goals for V1

Do not implement in V1:

- authentication;
- user accounts;
- multi-thread history browser;
- editing charts directly in the browser;
- CSV-based re-charting;
- dashboard builder;
- drag-and-drop chart configuration;
- persistence of frontend state beyond `localStorage`;
- frontend access to DuckDB or Postgres;
- frontend execution of generated Python;
- complex design system.

## 8. Backend API Assumptions

The frontend expects these backend endpoints.

### 8.1 `POST /query`

Request:

```json
{
  "thread_id": "optional-existing-thread-id",
  "message": "Show releases by decade",
  "debug": true
}
```

Response:

```json
{
  "thread_id": "thread-123",
  "run_id": "run-456",
  "response": "Generated a chart showing releases by decade.",
  "chart_artifact": {
    "artifact_id": "artifact-789",
    "artifact_type": "plotly_html",
    "url": "/artifacts/artifact-789/content"
  },
  "sql": "SELECT decade, COUNT(*) AS releases FROM release_unique_view ...",
  "dataframe_preview": [
    {"decade": 1980, "releases": 120},
    {"decade": 1990, "releases": 450}
  ],
  "route": {
    "complexity": "simple",
    "selected_model": "cheap_model"
  },
  "validation": {
    "valid": true
  }
}
```

The frontend should tolerate missing optional fields.

---

### 8.2 `GET /artifacts/{artifact_id}/content`

Returns the Plotly HTML artifact.

Expected headers:

```text
Content-Type: text/html
```

The frontend renders this URL in an iframe.

---

### 8.3 `GET /health`

Response:

```json
{
  "status": "ok",
  "duckdb_available": true,
  "postgres_available": true
}
```

The frontend may use this for a lightweight status indicator.

---

### 8.4 Optional: `GET /threads/{thread_id}`

Can be added later to reload previous conversation runs.

Not required for V1 if the frontend stores only the current conversation in local state.

## 9. Core User Experience

### 9.1 Initial screen

The user should see:

- app title;
- short description;
- suggested queries;
- query input;
- empty result panel or welcome message.

Example title:

```text
Discogs Analytics Agent
```

Example subtitle:

```text
Ask natural language questions about the Discogs releases dataset.
```

---

### 9.2 Query flow

1. User writes or selects a query.
2. Frontend calls `POST /query`.
3. Frontend sends current `thread_id` if present.
4. Backend returns response and artifacts.
5. Frontend stores returned `thread_id`.
6. Frontend appends user and assistant messages to chat.
7. Frontend renders chart artifact in result panel.
8. Frontend shows SQL and dataframe preview if available.

---

### 9.3 Continue conversation

The frontend stores the current `thread_id` in:

```text
localStorage.currentThreadId
```

Subsequent `/query` requests include that `thread_id`.

This allows follow-up questions like:

```text
Now only for UK.
```

---

### 9.4 New conversation

Button label:

```text
New conversation
```

Behavior:

- clear local chat messages;
- clear current chart/result panel;
- remove `currentThreadId` from `localStorage`;
- do not delete backend data;
- next `/query` creates a new thread.

Do not label this as “Clear history” unless backend deletion is implemented.

## 14. API Client

`src/api/client.ts`

Responsibilities:

- centralize `API_BASE_URL`;
- call `/query`;
- call `/health`;
- normalize artifact URLs;
- handle HTTP errors.

Environment variable:

```text
VITE_API_BASE_URL=http://localhost:8000
```

Example:

```ts
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL ?? "http://localhost:8000";

export async function sendQuery(payload: QueryRequest): Promise<QueryResponse> {
  const response = await fetch(`${API_BASE_URL}/query`, {
    method: "POST",
    headers: {"Content-Type": "application/json"},
    body: JSON.stringify(payload),
  });

  if (!response.ok) {
    throw new Error(`Query failed: ${response.status}`);
  }

  return response.json();
}

export function toAbsoluteArtifactUrl(url: string): string {
  if (url.startsWith("http")) return url;
  return `${API_BASE_URL}${url}`;
}
```

## 15. Artifact Rendering

### 15.1 Plotly HTML rendering

Render Plotly HTML in an iframe.

Component:

```tsx
<iframe
  src={absoluteArtifactUrl}
  sandbox="allow-scripts"
  title="Generated chart"
/>
```

Notes:

- Plotly HTML requires JavaScript, hence `allow-scripts`.
- Do not use `dangerouslySetInnerHTML`.
- Prefer iframe for isolation.
- Avoid `allow-same-origin` unless required.

If no artifact is available, show:

```text
No chart yet. Ask a question or run one of the suggested queries.
```

### 15.3 Artifact loading state

While query is running:

```text

```

Optionally show a skeleton chart card.

## 19. Error Handling

Frontend should handle:

- backend unavailable;
- query failure;
- missing artifact;
- artifact load failure;
- timeout;
- invalid response shape.

User-facing errors should be concise.

Examples:

```text

```


Do not expose raw tracebacks in the UI.

## 31. Security Considerations

- Do not use `dangerouslySetInnerHTML` for Plotly HTML.
- Render generated HTML in iframe.
- Use iframe sandbox.
- Do not expose backend internal filesystem paths if possible.
- Use artifact URLs instead of raw local paths.
- Do not show raw tracebacks in user-facing UI.
- Do not send API keys to frontend.
- All model/backend secrets stay only in backend environment.

## 35. Executive Summary for Coding Agent

Build a React + Vite + TypeScript frontend for the Discogs Analytics Agent. The frontend should call the existing FastAPI `/query` endpoint, preserve `thread_id` in localStorage, display chat messages, render Plotly HTML chart artifacts in an iframe, show suggested demo queries, show generated SQL and dataframe preview, and allow starting a new conversation. It must not access DuckDB, Postgres, ETL files or generated artifacts directly; it only uses backend API responses and artifact URLs. CSV-based browser rendering and chart editing are stretch goals, not V1 requirements.
